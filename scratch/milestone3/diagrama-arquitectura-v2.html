2Flujo del usuario del npm install al recibo verificado por un tercero
01

El developer instala el SDK y registra su agente: register_agent crea el PDA (uno por operador).

02

El agente ejecuta una acción → el SDK calcula el hash del payload y lo firma con Ed25519.

03

record_attestations sella hasta 100 recibos en una tx — finality <1s, <$0.001 por recibo.

04

El programa valida la firma (instructions sysvar) y emite AttestationIssued on-chain.

05

El indexer capta el evento por WebSocket y lo proyecta a Postgres para consultas eficientes.

06

Cualquier tercero verifica: Explorer, REST API o Solana Explorer — sin confiar en el operador. Opcional: x402 ($0.01) y Blinks para compartir.

3Frontera on-chain / off-chain — proyección, no duplicación respuesta a la revisión del M3
QuéOn-chain (fuente de verdad)Off-chain (proyección consultable)
Registro del agentePDA AgentAccount — operator, agent_id, policy_root, contadorFila en agents para búsquedas y stats
Recibo (atestación)Evento AttestationIssued: action_hash + firma Ed25519 + timestamp + tipoFila en attestations, indexada para filtros / paginación / breakdown por tipo
Payload crudoNunca — solo su hash (commitment)Opcional metadataUri si el operador quiere adjuntar contexto
Recuperación ante falloInmutable por consenso — no aplicaRe-indexable al 100% re-escaneando la cadena (probado: backfill de 26,000+ recibos tras endurecer el indexer)
Principio: on-chain guarda el mínimo verificable (la prueba); off-chain guarda la misma prueba proyectada para consultabilidad y UX. No es doble escritura de datos distintos: es una vista derivada que cualquiera puede reconstruir y auditar contra la cadena.
Riesgos:  costo/límites de RPC · resiliencia del indexer (catch-up 60s + watchdog + health real) · economía on-chain a alta frecuencia (batching ≤100/tx) · custodia de claves · consultabilidad depende del indexer (mitigado: reconstruible)
